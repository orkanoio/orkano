# Working on Orkano

You are helping build Orkano, an open-source, self-hosted PaaS that runs on Kubernetes. The project is being built by one person in their spare time. Optimize every suggestion for that reality.

## Always read these first
Before starting any task, read these in order:
1. `PRD.md` — what Orkano is, who it's for, what's in scope, what's out.
2. `PLANNING.md` — architecture, stack choices, milestones, risks, constraints.
3. `TASKS.md` — the current task list. The next unchecked item under the current milestone is the source of truth for what to work on.

If the user gives you a task that isn't on the list or that conflicts with the PRD's non-goals, stop and surface the conflict before writing code. Do not silently expand scope.

## Stay within scope
- The PRD's `Non-Goals` and `What Orkano Supports (v1 Scope)` sections are binding.
- If a "nice to have" idea comes up mid-task, add it to TASKS.md under the appropriate future milestone (or the Backlog) — do not implement it now.
- Prefer cutting features to slipping milestones. PLANNING.md calls out which v1 features are safe to cut (Redis/MongoDB, Nixpacks) and which are not (TLS, GitHub App integration, live logs, the security architecture).

## Update TASKS.md as you work
- Flip a task to `[~]` when you start it; flip it to `[x]` only after the task is fully complete (tests + docs included, where applicable).
- If a task turns out to be wrong or unnecessary, mark it `[-]` with a one-line reason next to it.
- If you discover a task needs to be split, split it in place — preserve the original ordering and add the new subtasks underneath.
- Do not rearrange milestones without explicit user confirmation.

## Decision principles (in priority order)
1. **Solo-maintainer-friendly.** Choose the option that is easiest to debug at 11pm with no help. Boring, mature tools beat clever ones.
2. **Open-source-friendly.** Prefer permissively licensed dependencies. Avoid anything that requires a paid API key to run locally.
3. **Kubernetes-native.** Don't reinvent what Kubernetes already does well (scheduling, secrets, retries). Wrap it; don't replace it.
4. **Self-hostable by default.** No required external SaaS dependency to run Orkano. Optional integrations are fine; mandatory ones are not.
5. **Honest abstractions.** UI concepts must map to real Kubernetes objects so users can graduate to `kubectl` without a migration.
6. **Small surface area.** Each new endpoint, table, or component should justify its existence. When in doubt, leave it out.

## Code style
- **Go:** idiomatic, `gofmt`, no unused exports, errors wrapped with `%w`, contexts threaded through.
- **TypeScript:** strict mode, no `any`, prefer functional components and hooks, colocate types with the code that owns them.
- **SQL:** hand-written queries via `sqlc`; no ORMs.
- **Tests:** prefer integration tests over heavily mocked unit tests, especially for anything touching the Kubernetes API.
- **Comments:** none, unless the *why* is non-obvious. Identifier names carry the *what*.

## When unsure
- Ask the user instead of guessing on anything that affects architecture, public API, or scope.
- For implementation details inside an already-agreed task, make the call and note the choice in the PR description.
- If a task in TASKS.md is ambiguous, propose a one-sentence clarification and confirm before starting.

## What not to do
- Do not introduce new top-level dependencies without flagging them.
- Do not add telemetry, analytics, or external network calls without explicit approval.
- Do not generate Markdown docs or design notes that the user didn't ask for.
- Do not rewrite working code "while you're in there." Touch only what the task requires.
- Do not amend or force-push without explicit instruction.

### Key functions (operator/internal/controller/app_controller.go)
- `Reconcile` renders Deployment+Service from `status.image`; INV-06 refuses non-digest-pinned images.
- `observeBuilds` derives `status.latestBuild`/`status.image` from Builds via the `spec.appName` field index (`buildAppNameIndex`); status only ever advances — pruned Builds never regress it. `newerBuild` orders by creationTimestamp, name tie-break.
- `setReady`/`failReady`/`updateStatus` — Ready condition + `observedGeneration`, written only on semantic change. Reasons: `WaitingForBuild`, `InvalidImage`, `ReconcileError`, `Progressing`, `Available`. `status.url` belongs to the Domain reconciler — never write it here.
- `mutateDeployment`/`mutateService` must stay no-ops against stored objects (idempotency is asserted in tests); never adopt foreign same-name objects.
- `SetupWithManager` registers the Build index + `Watches(Build, mapBuildToApp)`.

### Key functions (operator/internal/controller/domain_controller.go)
- `Reconcile` renders one Ingress per Domain (annotation `cert-manager.io/cluster-issuer` from the `--cluster-issuer` flag, default `orkano-platform` — M1.5's component deploy must create the ClusterIssuer under that name); never creates Certificates, ingress-shim does (ADR-0006). The Ingress renders even before the App exists, so cert issuance runs parallel to the first build.
- Certificates are read/watched as `unstructured` (deliberate: no cert-manager Go dep) — the Certificate CRD must exist before the operator starts. Certificate name = TLS secret = `tlsSecretName()` = `<domain>-tls`; `mapCertificateToDomain` inverts it.
- Conditions: `Ready` (reason precedence HostConflict > ReconcileError > AppNotFound/AppNotRoutable > CertificatePending > Available) + `CertificateReady` mirrored from the cert's Ready condition.
- Host conflicts: cluster-wide `spec.host` field index, oldest creationTimestamp wins, name tie-break (`olderDomain`); losers delete their own Ingress (also heals the create-race where a loser rendered first); `mapDomainToHostPeers` promotes the next-oldest on winner deletion.
- `updateAppURL` is the ONLY writer of `App.status.url`: `https://<host>` of the oldest live conflict-winning Domain of a Web app, else "". Reads the App via `APIReader` (uncached) — on the finalizer path a stale equality-skip would never heal. Finalizer `orkano.io/domain` exists solely for that recompute; Ingress cleanup is plain ownerRef GC.
- `mutateIngress` must stay a no-op against stored objects; `ingressClassName` deliberately untouched (cluster default).

### Key functions (operator/internal/controller/build_controller.go + operator/internal/registry)
- `BuildReconciler{Client, APIReader, ResolveDigest}` runs one Build as one Job (rendered by buildjob). Terminal phases (Succeeded/Failed) are CLOSED RECORDS — the controller never touches them again; retry = create a new Build. Every dead end is terminal on purpose (a non-terminal refusal would retry forever and pin `App.status.latestBuild`).
- `ensureJob` creates or adopts (annotations prove ownership; crash window between Create and the jobRef status write heals by adoption; AlreadyExists re-reads live via APIReader). Foreign same-name Job → terminal `JobConflict`, never adopted, never deleted. jobRef set + Job gone (confirmed via APIReader — the cache may lag our own create) → terminal `JobMissing`, no silent rebuild.
- `observeJob` derives phase from Job conditions: Failed→`failureReason` (DeadlineExceeded from the Job condition reason; OOMKilled/exit-code from pod container statuses listed UNCACHED — the pod informer lags the final statuses; scans InitContainerStatuses too so a static build's render-dockerfile init failure surfaces); Complete→`ResolveDigest` then Succeeded+image+completedAt written together (Succeeded with empty image would roll Apps to nothing); digest error → stays Running, condition reason `ResolvingDigest`, backoff retries. Pending only ever set when phase is ""/Pending (a cache-lagged Active=0 must not regress Running).
- The Job's BuildKit invocation comes from `buildjob.Compose(build)` (git context, Dockerfile path, image ref); `ensureJob` passes all three into `Render`, `observeJob` reuses `Compose(build).ImageRef` as the digest-resolution target (Compose is pure, the spec is immutable — recompute beats threading). Tag = commit (two Builds of one commit overwrite each other's tag, each pins its own digest). The old provisional `contextURL`/`imageRef` controller helpers are gone.
- Finalizer `orkano.io/build`: deleting a Build deletes its Job (Background propagation EXPLICITLY — the batch API's legacy default is orphan) — but only a Job the annotations prove ours.
- `registry.Resolver{Reader}` (uncached APIReader): HEAD `https://<host>/v2/<repo>/manifests/<tag>` → `Docker-Content-Digest` → `host/repo@sha256:…`. TLS root = ConfigMap `orkano-registry-ca` in orkano-builds fetched per call (rotation just works; RBAC = one resourceNames-pinned configmaps get, no informer). Transport per call with `DisableKeepAlives` (no stranded idle conns). Suite stubs ResolveDigest (`stubResolveDigest`/`setDigestStub`); the real resolver is tested against a local TLS server.

### Key functions (operator/internal/controller/postgres_controller.go)
- `PostgresReconciler{Client, Scheme, APIReader}` (M1.4, landed 2026-06-21) renders the v1 service catalog: a digest-pinned Postgres StatefulSet + headless Service + the connection Secret named EXACTLY `metadata.name` in orkano-apps (ADR-0014). No finalizer — cascade rides ownerReference GC + the StatefulSet's PVC retention policy.
- `Reconcile` order: validate name (`dns1035Label` regex — start with a letter, no dots, ≤63; the DNS-1123-subdomain CRD accepts looser names so a bad one is a user-visible `ProvisionFailed`, validation reconciler-side per ADR-0010) → resolve image from `postgresImages[version]` → check `storageSize >= minStorageSize` (1Gi floor) → ensureSecret → ensureService → ensureStatefulSet → reconcileStorage (grow/shrink) → set Ready. `failProvision` records the `ProvisionFailed` condition before returning the error (visible in `kubectl describe`; skips conflicts). `setPostgresReady`/`updateStatus` mirror the App controller (write status only on semantic change). Reasons: `Provisioning`/`ProvisionFailed`/`Available` (shares `reasonAvailable` with the App controller).
- The SQL identifier = name with `-`→`_` (`api-db`→`api_db`); used for POSTGRES_DB/USER and the DB/username Secret keys. `dns1035Label` guarantees it's a valid unquoted identifier AND a valid Service/StatefulSet name in one check.

A missing entry is `ProvisionFailed`, not a panic.

INV-03: the workload references the Secret by key, no value in the STS spec. Skips a no-op Update when data+ownerRef+labels already match.

### Key functions (operator/internal/githubapp — token.go + repo.go)
- `TokenSource` mints + caches GitHub App installation access tokens (token.go, M1.3 landed 2026-06-16) AND resolves repo+ref → commit SHA (repo.go, landed 2026-06-16) — the GitHub-App API surface the dispatcher consumes. Now WIRED into `main.go` via the dispatcher (the consumer). ZERO new deps: the App JWT (RS256 over a fixed 3-claim payload), the `POST /app/installations/{id}/access_tokens` exchange, and the three repo GETs are all hand-rolled with stdlib crypto/HTTP, mirroring `registry.Resolver`'s minimal-HTTP-against-a-documented-API philosophy. `go-github` is deliberately NOT pulled in (PLANNING blesses it, but hand-rolling these calls keeps the dep + audit surface minimal and matches the established pattern — noted in the commit).
- `InstallationToken(ctx, installationID) (string, error)`: returns a cached token if one is still >`expiryGuard` (2m) from expiry, else mints. Mint path is serialized under a `sync.Mutex` held ACROSS the HTTP exchange — deliberate: mints are ~once/hour, the operator is single-tenant, so simplicity beats singleflight (cost: a slow GitHub stalls callers up to the 15s `httpTimeout`). Cache is `map[int64]cachedToken` keyed by installation ID; failures (incl. a 201 with an empty token / no expiry) are NOT cached (next call retries). Rejects `installationID<=0` before any HTTP. The default HTTP client refuses redirects (`CheckRedirect`→`ErrUseLastResponse` — the JWT is a credential, don't forward it) and disables keepalives (mirrors `registry.Resolver`).
- `loadCredentials` reads the Secret per mint via an uncached `client.Reader` (the manager's APIReader) — so key rotation just works and the RBAC stays a single resourceNames-pinned `get`, no informer. Secret = `orkano-system/orkano-github-app` (overridable via `SecretRef`); data keys `app-id` (numeric) + `private-key.pem` (`parseRSAPrivateKey` accepts GitHub-native PKCS#1 AND PKCS#8). `buildAppJWT`: `iat` backdated 60s (clock skew), `exp` now+8m (under GitHub's 10m cap), `iss`=app id as a string; signed `rsa.SignPKCS1v15`+SHA256, base64url segments. Token + PEM are NEVER logged or persisted (INV-07); the non-201 error path surfaces the (truncated) GitHub error body — never the success body that carries the token.
- Exported config knobs (all default sanely): `Reader` (required), `SecretRef`, `BaseURL` (default `https://api.github.com`; override for the httptest stub / GHES), `HTTPClient`, `Now` (injectable clock). Exported constants `SecretNamespace`/`DefaultSecretName`/`AppIDKey`/`PrivateKeyKey` are the M1.5 component-deploy Secret contract. `DefaultSecretName` carries a `//nolint:gosec` for G101 (Secret object name, not a credential — same pattern as the receiver's `envSecret`).
- `mintAppJWT(ctx)` (token.go) is the shared App-JWT minter (loadCredentials + buildAppJWT), used by both `InstallationToken` and the repo.go installation lookup.
- `ResolveCommit(ctx, repo, ref) (sha, error)` (repo.go): given the App's repo + configured ref, returns the current 40-char HEAD SHA. Flow: `installationID` (App-JWT `GET /repos/{o}/{n}/installation`) → `InstallationToken` → if ref=="" `defaultBranch` (`GET /repos/{o}/{n}`.default_branch) → `GET /repos/{o}/{n}/commits/{ref}`.sha (validated `^[0-9a-f]{40}$`). App JWT used ONLY for the installation lookup; the installation token for the data reads (tests assert the separation per leg). Error classification drives the dispatcher's drop-vs-retry: the `ErrUnresolvable` sentinel wraps 404/422, a malformed/`..`-bearing ref, a non-owner/name repo, and a structurally-invalid 200 (non-SHA / id≤0 / empty default_branch) — all permanent (dispatcher drops the doorbell so a poison row can't head-of-line-block the FIFO queue); everything else (network/5xx/auth/rate-limit/unparseable body) is transient (retry). The ref has only a CRD length bound (no pattern), so it's `..`-guarded and `url.URL`-escaped before landing in the path (slashes preserved for `release/1.x`).
- `token_test.go` (white-box, no envtest/Docker): a `githubStub` httptest server that VERIFIES the App JWT signature against the test key + asserts claims (iss, exp-iat≤600, iat backdated) and a controller-runtime `fake` client for the Secret. Covers mint→cache-hit→refresh-near-expiry (request-count asserted), per-installation cache isolation, GitHub-error-not-cached, invalid id (no HTTP), and a credential-error table (missing/empty/non-numeric app-id, missing/garbage key, PKCS#1 ok, PKCS#8 ok, NotFound wrapped). RBAC grant is proven live by `rbac_matrix_test.go` (the `secrets[orkano-github-app] get` row).
- `repo_test.go` (white-box, httptest stub): default-branch vs explicit-ref paths, App-JWT-vs-installation-token auth on every leg, slash-in-ref escaping, the ErrUnresolvable table (incl. `..` and non-SHA), transient-500-not-dropped, and `splitRepo` cases.

### Key functions (operator/internal/dispatcher)
- The webhook queue consumer → Build creator (M1.3, landed 2026-06-16). A leader-only `manager.Runnable` (`NeedLeaderElection()`→true so exactly one replica drains the queue): `Start` polls every `PollInterval` (default 5s, `--dispatch-poll-interval`); `tick` computes a `budget` (`MaxConcurrentBuilds` − non-terminal Builds; default 5, `--max-concurrent-builds`) and processes up to that many deliveries, stopping early on an empty queue or any transient error (back off to the next poll). Cap is approximate at delivery granularity (a monorepo fan-out of one delivery can briefly exceed it).
- `processOne` claims one delivery (`Queue.ClaimNext`), calls `handle`, then finalizes: Ack (remove) on success or a permanent drop, Nack (leave for retry) on a transient failure. `handle` resolves `appsForRepo` (cluster-wide `List(App)` + case-insensitive repo match — one repo backs many Apps in a monorepo via subPaths), and per App: `GitHub.ResolveCommit(app.repo, app.ref)` → `createBuild`. Drop-vs-retry: no matching App → drop (the receiver allowlist gates repos; an allowlisted repo with no App is config, not retry); per-App `ErrUnresolvable` → skip THAT App (other monorepo Apps may resolve), drop only if ALL are unresolvable; any transient error → Nack the whole doorbell (idempotency makes the re-process safe). This is a queue-availability property: a poison row can never head-of-line-block the FIFO queue.
- `buildFromApp` snapshots `app.Spec.Source`/`app.Spec.Build` into an immutable Build pinned to the SHA; TimeoutSeconds left unset (CRD default applies); standalone (no ownerRef — App controller finds it via the `spec.appName` index; lifecycle independent of the App, matching how Builds are created elsewhere). `buildName(app, sha[:12])` is DETERMINISTIC → a duplicate delivery / a re-resolve to the same HEAD / a re-processed row all converge on one Build (`createBuild` treats AlreadyExists as success). This is the whole idempotency story: the doorbell is level-triggered — it always builds the CURRENT HEAD of the App's tracked ref, never a payload commit (INV-04), so intermediate pushes collapse. Long App names cap at 253 with a hash tail (mirrors `buildjob.JobName`). The dispatcher NEVER writes App status — `latestBuild`/`image` stay derived by the App controller's `observeBuilds`.
- `queue.go`: `Queue` interface + `PgxQueue` (production). `ClaimNext` begins a tx, runs `SET LOCAL idle_in_transaction_session_timeout = 0` (the tx is held across the GitHub round-trips, which can outlast a hardened/managed Postgres's idle-in-tx timeout; `SET LOCAL` reverts on commit/rollback, USERSET so the dispatcher role may set it), then `ClaimDelivery`. Returns a `*Delivery` whose `Ack` = DeleteDelivery + COMMIT and `Nack` = ROLLBACK (leave the row, release the lock); empty queue → (nil, nil). At-least-once delivery (lock held until finalize), made exactly-once by the deterministic Build name. Shutdown/panic are safe: at-least-once + idempotency cover a cancelled-ctx finalize or a crash (process death reclaims the connection).
- No new K8s RBAC: the operator's orkano-apps Role already grants `apps get/list/watch` + `builds create/list`; the DB uses the existing `orkano_dispatcher` role. Egress to GitHub + Postgres is open (orkano-system has no default-deny).
- Tests: `dispatcher_test.go` (fake client + `fakeQueue` + `fakeGitHub` keyed on `repo@ref`) — snapshot fidelity (full 40-char SHA, source/strategy copied), monorepo fan-out, partial-monorepo (one resolvable + one ErrUnresolvable → build the resolvable, drop), duplicate-delivery idempotency, no-App drop, unresolvable drop, transient Nack (+ re-queued), the concurrency cap (budget 0 → queue not polled), case-insensitive match, ref-forwarding, and `buildName`. Build ADMISSIBILITY (the snapshot is a valid Build per CEL) is proven separately by `examples_test.go` in real envtest, so the fake client here is sufficient for orchestration. `queue_test.go` (testcontainers, under the `orkano_dispatcher` role) — Ack/Nack tx semantics + SKIP LOCKED across an outstanding claim (deadline-guarded).
